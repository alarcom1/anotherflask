What is an Object in Python?. Almost everything in Python is considered as an object. An object has its own properties(attributes) and behavior(methods). A class is a blueprint of the objects or can be termed as object constructor for creating objects. One class can have many objects and value of properties for different objects can be different.

What is a Class in Python? Python is an object oriented programming language. Almost everything in Python is an object, with its properties and methods. A Class is like an object constructor, or a "blueprint" for creating objects.

What is a Namespace in Python? A namespace is a collection of names and the details of the objects referenced by the names. We can consider a namespace as a python dictionary which maps object names to objects. The keys of the dictionary correspond to the names and the values correspond to the objects in python. Or in other words, it is also the folders structure in your Python code environment. A class is life a file, a Namespace is like the folder.

The constructor is a method that is called when an object is created. This method is defined in the class and can be used to initialize basic variables. If you create four objects, the class constructor is called four times. Every class has a constructor, but its not required to explicitly define it.

A class is a user-defined blueprint or prototype from which objects are created. Classes provide a means of bundling data and functionality together. S - The Single-Responsibility Principle (SRP). A class should have one, and only one responsibility.

O - The Open-Closed Principle (OCP). You should not need to modify the code you have already written to accommodate new functionality, but simply add what you now need.

L - The Liskov Substitution Principle (LSP) - Functions that use pointers or references to base classes must be able to use objects of derived classes without knowing it

I - The Interface Segregation Principle (ISP) - Many client-specific interfaces are better than one general-purpose interface

D - The Dependency inversion Principle (DIP) - Abstractions should not depend on details. Details should depend on abstraction. High-level modules should not depend on low-level modules. Both should depend on abstractions
